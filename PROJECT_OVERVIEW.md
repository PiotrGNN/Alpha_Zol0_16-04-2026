# ZoL0 Current-State Technical Overview

## A. Executive Summary

ZoL0 is a KuCoin-only, PAPER-first trading-system codebase with runtime validation, telemetry, and readiness-gate tooling.

This repository contains source code, tests, and scripts for controlled PAPER validation. This document does not claim profitability, production readiness, or LIVE readiness. Current runtime/profitability readiness must be determined from fresh gate artifacts generated by the repository scripts, not from historical LEVEL status notes.

## B. Current operating model

- Default operating assumption: PAPER mode.
- Exchange scope: KuCoin only.
- LIVE mode is hard-gated and must not be inferred from code presence alone.
- Runtime outputs such as DBs, logs, reports, KPI CSVs, and diagnostics are generated artifacts and are not source-of-truth repository content.
- Controlled validation is centered on `scripts/controlled_kpi_run.py`, `scripts/run_paper_readiness_gate.py`, and related audit/reporting scripts.

## C. Proven project constraints

The following constraints are directly represented in the repository:

- `LOCKED_FACTS.md` records KuCoin-only, PAPER-only, and no-LIVE-impact constraints for the locked runner validation state.
- `tests/test_exchange_kucoin_only.py` scans runtime paths for non-KuCoin host/env tokens and verifies non-KuCoin platform registration is blocked.
- `security/live_guard.py` requires all LIVE arming inputs before LIVE can be considered armed: `LIVE=1`, `LIVE_ARMED=1`, valid `LIVE_APPROVAL`, `ZOL0_TOKEN`, and KuCoin API credentials.
- `core/db_models.py` disables SQLite fallback when `LIVE=1` by requiring an explicit `DATABASE_URL`.
- `core/MarketDataFetcher.py` blocks non-KuCoin API URLs and applies explicit controls for mock OHLCV usage in KuCoin PAPER contexts.
- The test suite contains PAPER runtime, KuCoin client, paper quote path, entry-gate, readiness, and LIVE-gate coverage.

## D. Main modules and their roles

- `core/BotCore.py`: main runtime loop and PAPER execution control path; includes entry classification, gate logic, lifecycle handling, telemetry emission, and shutdown/reconciliation logic.
- `core/MarketDataFetcher.py`: KuCoin market-data fetcher with URL validation, retry/throttling paths, and controlled mock-data behavior.
- `core/kucoin_client.py` and `core/kucoin_futures_client.py`: KuCoin spot/futures client utilities and request helpers.
- `core/OrderExecutor.py`: order-execution request construction and guarded exchange interaction paths.
- `core/PositionManager.py`: in-memory/persistence-facing position lifecycle management.
- `core/RiskManager.py`: risk checks and kill-switch integration.
- `core/db_models.py` and `core/db_utils.py`: persistence setup and DB write helpers, including LIVE/PAPER persistence policy.
- `scripts/controlled_kpi_run.py`: controlled PAPER KPI runner and diagnostic artifact producer.
- `scripts/run_paper_readiness_gate.py`: PAPER readiness gate orchestration and go/no-go report generation.
- `scripts/profitability_audit_scorecard.py`: profitability/economics audit scorecard generation from accepted artifacts.
- `security/live_guard.py` and `security/zero_trust.py`: LIVE arming and zero-trust helper logic.
- `strategies/`: strategy implementations used by runtime and tests.
- `models/`, `ai/`, `fl/`: model, AI, and federated-learning related modules present in the repo. Their existence is not a profitability claim.
- `tests/`: regression and contract coverage for KuCoin-only policy, PAPER runtime paths, database behavior, entry gates, readiness helpers, and selected audit/reporting scripts.

## E. Runtime readiness model

Readiness is artifact-driven.

The repository source alone does not prove that the system is ready for LIVE use or profitable PAPER operation. Current readiness must be established by running the relevant gate scripts and inspecting their output artifacts.

Current readiness evidence should come from:

- `reports/paper_readiness/*` generated by `scripts/run_paper_readiness_gate.py`.
- `artifacts/diagnostics/*`, `results/*`, and `tmp/*` generated by controlled validation scripts.
- Scorecards or audit outputs generated by `scripts/profitability_audit_scorecard.py` and related scripts.
- Test output from `pytest -q`.

Promotion or readiness claims require fresh artifacts showing the gate status and economics status. If fresh artifacts are absent, stale, missing, or failing, readiness is unconfirmed.

## G. Exact run/test commands

Run a controlled PAPER command from the repository root:

```powershell
Push-Location Alpha_Zol0-lvl_5-main
python scripts/controlled_kpi_run.py --variant-only before --before-min 5
Pop-Location
```

Run tests from the repository root:

```powershell
python -m pytest -q
```

Run a compile check from the repository root:

```powershell
python -m py_compile Alpha_Zol0-lvl_5-main/core/BotCore.py
```

Run the PAPER readiness gate from the nested project directory:

```powershell
Push-Location Alpha_Zol0-lvl_5-main
python scripts/run_paper_readiness_gate.py
Pop-Location
```

## H. Final repo-grounded description

ZoL0 is a KuCoin-only trading-system repository focused on PAPER-first runtime validation. It contains Python runtime modules, KuCoin client integrations, strategy/model components, telemetry and persistence helpers, controlled KPI runners, readiness-gate scripts, audit utilities, and regression tests.

The project is currently described as a validation and hardening codebase, not as a proven profitable or LIVE-ready trading product. Runtime readiness and profitability status must be determined from fresh gate artifacts and test results. Historical LEVEL and vision documents may describe prior ambitions or milestones, but they are not authoritative evidence of current readiness.
